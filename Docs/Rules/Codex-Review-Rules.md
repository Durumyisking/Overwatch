# Codex Review Rules

## Purpose
- Use this document for both pre-implementation responsibility checks and post-implementation code review.
- The goal is to prevent responsibility drift, reduce unnecessary dependencies, and keep useless code from accumulating.

## Pre-Implementation Prompt
작업을 시작하기 전에 먼저 "이 변경이 현재 함수의 흐름과 맥락에 맞는가"를 점검한다.

목표:
- 클래스 이름만 보고 역할을 고정해서 판단하지 않는다.
- 지금 수정하려는 함수의 이름, 호출 흐름, 호출자 기대, 현재 맥락에 맞는 책임 배치를 선택한다.
- 가장 빨리 붙는 코드가 아니라, 가장 자연스럽고 덜 놀라운 위치에 로직을 둔다.
- 불필요한 의존성과 임시 코드를 늘리지 않는다.

반드시 먼저 답할 것:
1. 지금 수정하려는 함수/메서드는 현재 흐름에서 어떤 역할을 하는가?
2. 이 함수를 호출하는 쪽은 이 함수가 어디까지 해줄 거라고 기대하는가?
3. 내가 넣으려는 로직이 그 함수 이름과 맥락에서 자연스러운가, 아니면 숨은 부작용인가?
4. 이 로직은 현재 위치보다 더 자연스러운 진입점이 있는가?
5. 지금 추가하려는 include / direct reference / member access가 정말 필요한가?
6. direct call 대신 delegate / event / interface / 기존 흐름 연결로 풀 수 있는가?
7. 이 코드는 "지금 빨리 동작하게 하려고" 넣는 것인가, 아니면 구조적으로 맞는 위치인가?

판단 원칙:
- 어떤 클래스든 맥락에 맞으면 여러 역할을 할 수 있다.
- 하지만 특정 함수는 그 함수의 이름과 호출 흐름이 암시하는 책임 범위를 넘지 않아야 한다.
- 문제는 "이 클래스가 이 역할을 하면 안 된다"가 아니라, "이 함수 맥락에서 그 역할이 자연스러운가"이다.
- 함수 이름보다 더 큰 책임을 숨겨 넣지 않는다.
- 호출자 입장에서 예상하기 어려운 저장, 바인딩, 상태 전이, 도메인 해석은 더 자연스러운 위치가 있는지 먼저 본다.
- 입력/UI/도구 성격의 코드가 도메인 규칙을 직접 해석하려고 하면 한 번 더 의심한다.
- 반대로 도메인 객체가 현재 흐름의 자연스러운 수신자라면 그쪽으로 책임을 모은다.
- 새 bool, helper, wrapper, cache, bridge 코드는 정말 필요한 경우만 추가한다.

강제 체크:
- 이 함수 이름만 보고 다른 엔지니어가 지금 넣은 로직을 예상할 수 있는가?
- 이 로직이 여기 있으면 함수가 더 명확해지는가, 아니면 더 숨겨지는가?
- 더 적은 의존성으로 같은 문제를 해결할 수 있는가?
- 지금 구조가 "맞는 구조"인가, 아니면 "임시로 붙인 구조"인가?

출력 전에 스스로 확인:
- 나는 지금 클래스 고정관념으로 판단하고 있나, 아니면 함수의 흐름과 맥락으로 판단하고 있나?
- 이 변경은 호출자에게 덜 놀랍고, 책임 경계가 더 분명한가?

## Post-Implementation Prompt
구현이 끝난 뒤에는 "이 코드가 실제로 잘 분리되었는가"를 하네스 관점으로 검토한다.

하네스 관점:
- 이 로직을 주변 시스템 없이도 설명할 수 있는가?
- 입력 -> 이벤트 -> 도메인 해석 -> 상태 저장 흐름이 분리되어 있는가?
- 특정 객체를 가짜로 바꿔도 핵심 로직이 유지되는가?
- 도메인 로직이 UI/입력/컨트롤러에 박혀 있지 않은가?

반드시 점검:
1. 새로 생긴 include 중 불필요한 것은 없는가?
2. direct reference를 delegate/event로 바꿀 수 있는데 그대로 둔 것은 없는가?
3. 호출되지 않는 helper, 임시 wrapper, no-op 함수, 중복 분기, 중복 상태 동기화가 남아 있지 않은가?
4. 이름은 단순한데 내부 책임이 과한 함수가 없는가?
5. BeginPlay/EndPlay, Attach/Detach, Bind/Unbind가 짝이 맞는가?
6. 저장되는 상태와 화면상 상태가 따로 놀 가능성이 없는가?
7. 미래 확장용이라고 넣은 speculative code가 없는가?

불필요 코드 제거 원칙:
- 지금 쓰이지 않는 helper는 추가하지 않는다.
- 한 번만 쓰는 wrapper는 제거를 우선 검토한다.
- 호출 경로가 없는 바인딩/해제 코드는 지운다.
- 같은 정보를 두 군데 저장하면, 꼭 필요한 경우가 아니면 하나로 줄인다.
- 설명되지 않는 임시 변수/멤버/플래그는 남기지 않는다.
- "나중에 쓸 수도 있음" 수준의 코드는 넣지 않는다.

최종 정리 시 반드시 보고:
- 남긴 코드 중 왜 필요한지 설명 가능한 것만 남겼는가?
- 이번 변경에서 같이 제거할 수 있었던 죽은 코드가 있었는가?
- 구조를 단순하게 만들었는가, 아니면 기능만 얹었는가?
